**VERDICT: REVISE**  
**판정 라벨: PRODUCTION_UNSAFE for public release until data artifacts are removed.**  
가장 먼저 고칠 1개: **A군 Sharadar 직접파생 파일을 `git rm --cached` + `.gitignore` 처리하고, public repo라면 history purge 여부를 별도 결정해라.**

1. **[발견] A군은 빼라.**
왜 실패: `sp500_pit.csv`, `broad_top1500.csv`, cap-weight/equal-weight/universe 파일은 Sharadar 원천의 PIT membership, 시총, 가격 파생물이다. “원천 CSV가 아니니까 괜찮다”는 방어가 약하다.  
임팩트: public repo에서 라이선스/redistribution 회색지대가 아니라 **라이선스 리스크 본체**다.  
수정안: **A군 전부 git 제거.** 재현은 “Sharadar 구독자가 local에서 생성”으로만 남겨라.

2. **[발견] B군은 남기되, aggregate evidence만 남겨라.**
왜 실패: negative-result 증거 공개가치는 있다. 하지만 날짜별 return/NAV/series 형태 CSV는 vendor-derived time series로 보일 수 있다.  
임팩트: 논문/리포트용 “결론 수치”와 “재배포 가능한 데이터셋”이 섞인다.  
수정안: `b2b5_factor_ic.csv`, `b2b5_broad_result.csv`, `regression_report.md` 같은 **요약 통계는 유지 가능**. `model_bakeoff.csv`, `multi_anchor.csv`처럼 날짜별 성과 series는 repo 밖 artifact로 빼거나 markdown 요약표로 축약.

3. **[발견] C군은 빼라.**
왜 실패: paper NAV, book ledger, target JSON, forward logs는 전략 운용 artifact다. 라이선스보다 더 단순하게, public repo에 committed될 필요가 없다.  
임팩트: 재현성보다 혼선이 크다. “paper/live/backtest” 라벨 오류가 이미 반복된 repo에서 산출물 커밋은 감사 리스크다.  
수정안: `phase1a/outputs/**`, `*_nav.csv`, `*_book.json`, `book_ledger.csv`, `forward_log.csv`, `production_target.json`, `prices.csv`는 기본적으로 gitignore. 필요한 증거는 report에 static summary만.

4. **[발견] `forward/prices.csv`는 Sharadar 위반은 아니지만 빼라.**
왜 실패: yfinance/Yahoo 공개 ETF 가격이면 Sharadar 약관 문제는 아니다. 하지만 Yahoo 약관은 자동 수집 금지 및 데이터로 대체/경쟁 DB를 만드는 사용을 제한하고, 명시 허가 없이는 서비스/content의 재사용·배포·파생 상업 exploitation을 금지한다는 구조다. Yahoo Terms lines 179-188 참고: automated collection 제한 및 reproduce/distribute/create derivative works 제한이 있다.   
임팩트: “무료 공개가격이라 git에 둬도 됨”은 틀림. public repo 가격 CSV는 불필요한 ToS 표면이다.  
수정안: **가격 cache 전부 제거.** runner가 로컬에서 다시 받게 해라.

5. **[발견] `rm --cached`만으로는 future clean이지 past clean이 아니다.**
왜 실패: GitHub 문서는 민감 데이터 제거에 history rewrite가 필요할 수 있고, rewrite는 부작용이 크다고 명시한다. 또한 force-push 후에도 clones/forks/PR refs/cache에 남을 수 있다고 설명한다. ([docs.github.com](https://docs.github.com/en/authentication/keeping-your-account-and-data-secure/removing-sensitive-data-from-a-repository))  
임팩트: public repo에 이미 올라간 A군을 “완전히 없앴다”고 말하면 거짓이다.  
수정안: **비공개/내부 repo면 `rm --cached` 충분. public이고 A군이 이미 노출됐으면 filter-repo 검토.** 단 secret은 아니므로 GitHub Support까지 가는 수준은 과할 수 있다.

6. **[발견] negative-result 공개가치를 데이터 파일로 보존하려는 전제가 위험하다.**
왜 실패: evidence는 “숫자와 방법”이면 된다. vendor-derived CSV 원장을 public에 둬야만 negative-result가 믿긴다는 건 과한 전제다.  
임팩트: 라이선스 리스크를 감수하면서도 검증성은 별로 안 오른다.  
수정안: 코드 + synthetic/sample schema + report tables + “requires Sharadar subscription to regenerate”가 맞다.

**파일군 판정**
- **A: 제거.** public git 유지 반대.
- **B: 부분 유지.** aggregate summary만 유지, 날짜별/재구성 가능한 series는 제거.
- **C: 제거.** outputs/cache/ledger/positions/forward logs는 git artifact가 아니다.
- **`forward/prices.csv`: 제거.** Sharadar 위반은 아니지만 Yahoo/yfinance 재배포 ToS 표면이라 public repo에 둘 이유 없음.

**실행 기준**
- 최소 조치: `git rm --cached` + `.gitignore`.
- public에 이미 push된 A군까지 지우려면: `git filter-repo`로 history purge 검토.
- secret/PII/key가 깨끗한 상태라면, purge는 “보안 필수”가 아니라 “라이선스 노출 축소” 결정이다. 그래도 A군이 public에 있었으면 나는 purge 쪽이다.
