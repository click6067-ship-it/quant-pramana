**≤6 Sharpest Objections**

1. **[발견] “실제 작동”을 24/7 라이브로 과장할 가능성**  
왜 실패: 현재 WSL cron은 PC 의존. “24/7”이라고 쓰면 구현 신뢰를 스스로 깎는다.  
임팩트: 작동성 평가에서 정직성 손상.  
수정안: “daily unattended paper runner on local cron; cloud 24/7 not claimed”로 박고, append-only logs와 fail-closed behavior를 제출.

2. **[발견] negative-result를 실패담으로만 두면 고득점 프레임이 안 됨**  
왜 실패: “알파 못 찾음”은 결론이고, 평가 대상은 왜 그 결론이 믿을 만한가다. null result 문헌도 비유의 결과 자체가 효과 부재 증거는 아니며 적절한 설계·해석이 필요하다고 지적한다. ([arxiv.org](https://arxiv.org/abs/2305.04587?utm_source=openai))  
임팩트: 교수는 “그래서 뭘 배웠나?”로 본다.  
수정안: “false alpha rejection machine”으로 프레임: look-ahead 발견, RVOL leakage 수정, next-bar, PIT, baseline, kill criteria.

3. **[발견] 8세대 서사는 과밀하다**  
왜 실패: v1~A2를 전부 자세히 넣으면 decision quality보다 산만함이 보인다.  
임팩트: 고민·의사결정 과정 점수 하락.  
수정안: 세대별 1줄 표: `Hypothesis / Test / Failure Mode / Decision / Artifact`. 본문은 turning point 5개만.

4. **[발견] “TQQQ 못 이김”을 평가 기준으로 들고 가면 자멸**  
왜 실패: TQQQ는 레버드 베타 benchmark라 학술 프로젝트의 합리적 기준이 아니다. active large-cap 펀드도 S&P 500 초과가 장기적으로 어렵다는 SPIVA 계열 결과가 반복된다; 2025에도 대형 active fund 79% underperform, 10년 기준 large-cap mutual fund underperform 84.3%라는 보도가 있다. ([kiplinger.com](https://www.kiplinger.com/investing/why-your-portfolio-needs-more-than-just-an-sp-500-etf?utm_source=openai))  
임팩트: 교수는 “비현실적 목표 설정”으로 볼 수 있다.  
수정안: benchmark ladder를 `SPY / QQQ / static 4-sleeve / naive leverage`로 분리하고, “TQQQ 초과 실패”는 humility note로만 둬라.

5. **[발견] backtest 수치보다 overfitting 방어가 핵심인데, 제출본이 수익률 중심이면 역효과**  
왜 실패: 금융 backtest는 configuration search로 쉽게 오염된다. Carr & López de Prado는 여러 historical simulation으로 rule을 calibration하는 행위 자체가 overfitting에 기여한다고 본다. ([arxiv.org](https://arxiv.org/abs/1408.1159?utm_source=openai)) Deflated Sharpe도 selection bias, multiple testing, non-normality 보정을 위해 나온 도구다. ([en.wikipedia.org](https://en.wikipedia.org/wiki/Deflated_Sharpe_ratio?utm_source=openai))  
임팩트: 좋은 숫자보다 “숫자를 안 믿은 이유”가 더 강한 프로젝트다.  
수정안: DSR/PBO/kill criteria/forward-only gates를 전면 배치.

6. **[발견] 재현성 패키지가 없으면 ‘작동’ 점수는 대시보드 미학으로 전락**  
왜 실패: 연구 재현성 문제는 흔하다. 한 재현성 조사에서는 충분한 정보와 낮은 재현 장벽을 가진 논문이 13%뿐이었다. ([arxiv.org](https://arxiv.org/abs/2305.01633?utm_source=openai))  
임팩트: 솔로 프로젝트일수록 “내 컴퓨터에선 됨”으로 보이면 감점.  
수정안: `README_SUBMISSION.md`, frozen sample data, one-command run, expected hash/row counts, dashboard snapshot을 묶어라.

**핵심쟁점 판정**
(a) 24시간 라이브데모 vs 재현가능 백테스트+로그: **후자 우선.** 라이브데모는 보조 영상/스크린샷이면 충분. 24/7을 주장하지 마라.

(b) negative-result 고득점 프레임: **“알파 실패”가 아니라 “가짜 알파를 죽이는 검증 OS”로 제출.** 성공 기준은 수익이 아니라 leakage 방지, baseline discipline, kill decision, forward runner 작동이다.

(c) 8세대+딥리서치 압축: **연대기 금지, 의사결정 DAG로 압축.** 각 세대는 “왜 다음 세대로 갔는가”만 남기고, 숫자는 decision을 바꾼 것만 넣어라.
