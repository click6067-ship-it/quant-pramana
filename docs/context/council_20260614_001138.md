**VERDICT: SHIP-public, with hygiene REVISE.**  
판정 라벨: `RESEARCH_ONLY` for trading content, `PRODUCTION_UNSAFE` for capital, **public repo는 유지**.

**결론:** private 전환은 과잉방어다. 숨길 알파가 없다. PRAMANA의 실제 자산은 “돈 되는 신호”가 아니라 “돈 안 되는 신호를 죽인 검증 OS”다. 이건 공개했을 때 포트폴리오/교육/신뢰 신호가 크고, private로 숨기면 오히려 “검증된 알파가 있는 척”하는 나쁜 프레이밍이 된다.

**≤6 sharpest objections**

1. **[발견] GPT의 “철학 공개가 손실” 논리는 핵심 자산을 오판함**  
   왜 실패: 공개되는 건 next-bar/PIT/kill/review 규율이지 실행 가능한 검증 알파가 아니다.  
   임팩트: private 전환은 평판 이득을 버리고, 없는 IP를 보호하는 비용만 냄.  
   수정안: public 유지. README 첫 줄에 “No verified alpha / paper only / validation OS” 박아라.

2. **[발견] 진짜 손실은 IP가 아니라 secret·개인정보·벤더 권리 위반**  
   왜 실패: 공개 repo의 리스크는 코드 구조보다 키/로그/원자료/계좌정보 누출이다. GitGuardian 계열 리포트는 2025년에 GitHub secret leak가 약 2,900만 건까지 갔다고 보고한다. AI-assisted code의 leak rate도 높다. ([techradar.com](https://www.techradar.com/pro/security/over-29-million-secrets-were-leaked-on-github-in-2025-and-ai-really-isnt-helping?utm_source=openai))  
   임팩트: private/public 논쟁보다 “public-safe audit” 미실시가 실제 사고면이다.  
   수정안: 공개 유지 전 `gitleaks`/GitHub secret scanning/히스토리 스캔. `.ndl_key`, raw Sharadar, positions, account NAV 상세, cron 경로, 로컬 절대경로 제거.

3. **[발견] public→private는 이미 공개된 것을 되감지 못함**  
   왜 실패: GitHub는 public repo를 private로 바꿔도 기존 public forks는 private가 되지 않고 분리된다고 문서화한다.   
   임팩트: 이미 노출된 민감정보가 있었다면 private 전환은 치료가 아니라 가림막이다.  
   수정안: 민감정보는 rotate + history purge + fork 대응. GitHub도 fork에 남은 민감 커밋은 계속 접근 가능하다고 경고한다. 

4. **[발견] private 전환은 보안 기능도 일부 잃을 수 있음**  
   왜 실패: GitHub 문서상 private 전환 시 Free/일부 플랜에서는 Pages가 내려가고, code scanning 등 Advanced Security 기능이 private에서 제한될 수 있다. public 전환 시에는 Advanced Security 기능 접근이 생긴다고 문서화되어 있다.   
   임팩트: “private=더 안전”이 자동 성립하지 않는다.  
   수정안: private가 아니라 public + secret scanning + push protection + clean `.gitignore`.

5. **[발견] 미래 알파 가능성을 이유로 지금 통째 private는 범위 오류**  
   왜 실패: 미래 알파는 현재 repo 전체가 아니라 특정 모듈/데이터/실행 로그에 생긴다.  
   임팩트: 지금 통째 private로 바꾸면 negative-result archive의 신뢰성과 교수 제출/포트폴리오 효과를 날림.  
   수정안: 미래에 양의 forward alpha가 나오면 `alpha_private/` 또는 별도 private repo로 분리. public repo에는 사전등록/결과/비민감 요약만 남겨라.

6. **[발견] 공개의 손실은 ‘경쟁자가 베낀다’가 아니라 ‘너를 오해한다’다**  
   왜 실패: A2/TQQQ/Attack Book이 public에서 “투자 조언/검증 전략”처럼 읽히면 평판 리스크가 생김.  
   임팩트: 알파 도난보다 더 현실적인 손실은 과장된 trading repo로 보이는 것.  
   수정안: README, dashboard, report 상단에 `PAPER ONLY`, `NO LIVE`, `NO VERIFIED ALPHA`, `NOT INVESTMENT ADVICE`, `Sharadar raw not redistributed` 명시.

**가장 먼저 고칠 1개:**  
README/루트 문서에 **public-safe disclosure boundary**를 추가해라: 공개 가능 = 코드·검증 프로토콜·negative results·비민감 대시보드. 공개 금지 = API key, Sharadar raw, broker/account data, live positions, proprietary future alpha module.

내 판정은 **public 유지**다. 단, 공개 전 보안/라이선스/프레이밍 audit 없이는 `REVISE`.
