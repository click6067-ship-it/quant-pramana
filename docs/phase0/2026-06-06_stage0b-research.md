## 8. paper-to-live drift 프로토콜 초안

> 목적: paper(모의)와 live 간 체결·비용·데이터 괴리를 *사전 정량화*하고 drift 임계 초과시 자동 게이트. KR-특수 최대 원천 = **NXT/SOR 분할 + 모의 NXT 미지원 가능성**.

1. **체결 모델 정합 (최우선)**: 백테스트·paper·live 모두 **KRX+NXT 합산 SOR 라우팅** 가정으로 통일. 단일거래소 가정 금지. NXT 저수수료가 비용모델을 낙관편향시키지 않도록 보수적 라우팅 가정 명시.
2. **모의=live semantics 괴리 점검 (G2 리스크)**: 모의투자(vps)가 NXT/SOR을 지원하는지 *개인 계정 실측*. 미지원이면 paper 체결이 KRX 단일거래소로만 일어나 live와 구조적 괴리 → drift 프로토콜의 1순위 점검항.
3. **비용 reconciliation**: STT(2025 0.15%/2026 0.20%, break point) + 슬리피지(square-root 충격) + 거래소수수료를 paper/live 양쪽 동일 비용모델로. ρ(비용모델-엔진수익) 모니터.
4. **레이트/인프라 한도 반영**: REST 20/s(실전)·모의 2~5/s[실측], 웹소켓 41종목/세션, 토큰 1분당 1회. 수집실패율 모니터(70종목 175/s→75% 실패 사례).
5. **데이터 정합 게이트**: live 데이터를 acceptance suite(§4)로 매일 통과 검증. CA·상폐·정정·PIT 멤버십을 공시 oracle 대조.
6. **drift 임계 veto**: paper vs live의 (체결가 갭·실현비용·신호 reproduction) 일별 추적, 임계 초과시 결정론 리스크엔진이 신규 진입 veto(자동).
7. **python-kis push 불신**: 웹소켓 push 신뢰불가(open 이슈) → REST 폴링으로 잔고·체결 reconcile.
8. **번들 검증**: NautilusTrader code-parity는 *벤더 주장* → 자체 parity 테스트 작성(bit-for-bit replay 검증).
